## 目的 (Purpose)

> 提供标准的Git代码提交流程，确保每次提交前敏感信息都被替换为占位符，提交信息格式规范，提交后敏感信息及时恢复。通过标准化的提交流程，防止敏感信息泄露到代码仓库，并确保提交信息格式规范。


## 成功标准 (必须满足所有要求)
- 敏感信息已替换：提交前所有敏感信息(数据库密码、Auth Secret、API Key)已替换为占位符
- 提交信息规范：提交信息包含一行总结和详细说明（1、2、3...点）
- 代码已提交：所有修改已添加到暂存区并成功提交到远程仓库
- 敏感信息已恢复：提交完成后所有敏感信息已恢复为实际值
- 流程完整：按照标准流程完成所有步骤

## 范围边界
- 替换敏感信息为占位符
- 添加所有修改到暂存区
- 使用规范的提交信息格式提交代码
- 恢复敏感信息
- 推送到远程仓库

## 本技能不负责
- 编写代码
- 解决代码冲突
- 代码审查
- 代码测试

## 转交点
- 当代码提交并上报后，技能就完成了。如果用户需要代码审查或测试，请移交给相应的技能。

## 使用场景
- 日常开发提交：开发完成后，按照标准流程提交代码
- 修复Bug提交：修复Bug后，按照标准流程提交代码
- 新功能提交：新功能开发完成后，按照标准流程提交代码
- 紧急修复提交：紧急问题修复后，按照标准流程快速提交代码

## 行为 (Behavior)

### 步骤1：替换敏感信息为占位符
> 在提交代码之前，必须将以下文件中的敏感信息替换为 `${}` 占位符：
1. application-dev.yml
2. application-pro.yml
3. application.yml
4. 其他可能的配置文件

### 步骤2：添加所有修改到暂存区

执行命令：
```bash
git add .
```

### 步骤3：提交代码

> 使用规范的提交信息格式，提交信息必须包含：
- 一行总结（简短描述本次提交的主要内容）
- 详细说明（列出 1、2、3... 点，说明具体修改内容）

**如果不知道本次修改的主要目的, 则从本次需要提交的文件中猜出主要修改的内容**

> 提交信息格式示例：

```
修复服务端包名导入错误

1. 修复 ChatController.java 中的包导入错误
2. 修复 AuthController.java 中的包导入错误
3. 修复 FriendController.java 中的包导入错误
4. 修复 GroupController.java 中的包导入错误
5. 批量替换所有 Java 文件中的错误包名
```

> 提交命令：

```bash
git commit -m "提交总结

1. 修改点1
2. 修改点2
3. 修改点3"
```

> 注意事项：
- 提交信息不能只有一行，必须包含总结和详细说明
- 详细说明中列出 1、2、3... 点，但不要过多（建议 3-5 点）
- **不要在提交信息中包含"替换配置文件中的敏感信息为占位符"等内容，这是固定行为，无需说明**

### 步骤4：恢复敏感信息

> 提交完成后，将之前替换的占位符恢复为实际的敏感信息：

4.1 恢复 application-dev.yml

- `password: ${DB_PASSWORD}` → `password: venus@911NASA`

4.2 恢复 application-pro.yml

- 保持占位符不变（生产环境应使用环境变量）

4.3 恢复 application.yml

- 保持占位符不变（应使用环境变量）

步骤5：推送到远程仓库（可选）

执行命令：
```bash
git push
```


## 示例 (Examples)
### 1. 示例 1：日常开发提交
> 用户完成了一个新功能的开发，需要提交代码。
执行流程：
1. 替换配置文件中的敏感信息为占位符。
2. 执行 git add . 添加所有修改。
3. 使用规范的提交信息格式提交代码。
4. 恢复配置文件中的敏感信息。
5. 执行 git push 推送到远程仓库。

```
添加用户头像上传功能

1. 新增用户头像上传接口
2. 添加头像图片压缩处理
3. 更新用户资料页面UI
4. 添加头像预览功能
```

2. 执行 git add . 添加所有修改。
3. 使用规范的提交信息格式提交代码。
4. 恢复配置文件中的敏感信息。
5. 执行 git push 推送到远程仓库。

提交信息示例：

## 附录 (Appendix)

### 1. 常见错误
1. 忘记替换敏感信息：会导致敏感信息泄露到代码仓库。
2. 提交信息不规范：只有一行总结，缺少详细说明。
3. 忘记恢复敏感信息：导致本地开发环境无法正常运行。
4. 提交信息过多：详细说明中列出过多要点，建议控制在 3-5 点。

### 2. 敏感信息清单
- 数据库账号密码。
- Auth Secret。
- API Key。
- 其他任何密钥、密码、令牌等。
